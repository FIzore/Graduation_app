## 1. 前端架构与基础能力

### 1.1 技术栈
- **框架**: Uni-app (Vue 3 `<script setup>` 组合式 API) + TypeScript
- **网络层**: 统一 `utils/request.ts` 封装，自动注入 JWT Bearer Token，超时 10s
- **WebSocket**: 单例引擎 `utils/websocket.ts`，指数退避重连 (1s→32s)、待发送队列、手动关闭隔离
- **状态管理**: `reactive()` 驱动的 `conversationStore`（会话仓库），无 Pinia/Vuex 引入
- **样式**: uView Plus (uni-icons) + SCSS，Flex 布局为主

### 1.2 统一拦截器（`request.ts`）
- **成功判定**: `200` 或 `201` 返回码直接 resolve
- **401 处理**: 清理 token → Toast 提示 → 1.5s 后 `reLaunch` 到登录页
- **错误优先级**: `403/400/429/500` 优先消费后端返回的 `msg`/`message` 字段，仅无后端文案时才回退通用提示
- **请求超时**: 10s，超时后提示"网络通信错误"
- **Token 注入**: 每次请求从 Storage 同步读取

### 1.3 WebSocket 引擎（`websocket.ts`）
- **单例模式**: 全局唯一连接实例，通过 `useWebSocket()` 获取操作句柄
- **指数退避重连**: 基础延迟 1s，最大 32s，`manualClose` 标记阻止无效重连
- **待发送队列**: 连接中断期间的消息进入 `pendingQueue`，连接恢复后自动冲刷
- **事件系统**: Pub/Sub 模式 (`on/off/emit`)，支持多监听者
- **生命周期**: `connect()` 在 `App.vue onLaunch` 调用；`close()` 在登出时重置一切

### 1.4 会话仓库（`conversationStore`）
- **一物一聊模型**: 键名为 `${otherId}_${itemId}`，确保与同一用户围绕不同物品的对话互不串扰
- **未读聚合**: `unreadTotal` 由 `Object.values(conversations).reduce` 实时计算
- **JWT 解码**: 从 Token payload 解析 `user_id`，缓存于 `cachedUserId`，避免重复解析

### 1.5 页面体系
| 页面 | 路径 | 认证要求 |
|---|---|---|
| 首页 | `pages/index/index` | 公开（OptionalAuth） |
| 搜索页 | `pages/search/search` | 公开 |
| 物品详情 | `pages/item/detail` | 公开 |
| 发布页 | `pages/item/post` | 私有 |
| 消息列表 | `pages/message/message` | 私有 |
| 聊天室 | `pages/chat/room` | 私有 |
| 个人中心 | `pages/profile/profile` | 公开（未登录展示登录入口） |
| 我发布的 | `pages/profile/published` | 私有 |
| 我的预约 | `pages/profile/appointments` | 私有 |
| 登录/注册 | `pages/auth/login` | 公开 |

### 1.6 自定义底部导航栏（`custom-tabbar.vue`）
- 完全替代微信原生 TabBar，支持 `home/message/post/me` 四入口
- 消息入口叠加未读红色角标（`unreadCount` props）
- `fixed` 定位 + `env(safe-area-inset-bottom)` 适配全面屏

## 3. 遗留体验债与已知缺陷

### 3.1 WebSocket 断连无用户感知
**问题归属**: 前端问题

**位置**: `utils/websocket.ts` + `pages/chat/room.vue`

**现象**: WebSocket 意外断开后，引擎在后台静默重连（指数退避），聊天室界面没有任何连接状态指示器。用户在断连期间发送消息会进入 `pendingQueue`，消息看起来已发出（乐观渲染到聊天列表）但实际对方收不到，直到重连成功后才冲刷队列。用户完全感知不到"当前连接中断、消息可能延迟"。

**严重程度**: 中 | **影响**: 用户可能误以为消息已送达，导致沟通错位。

### 3.2 Token 过期导致表单数据丢失
**问题归属**: 全栈问题

**位置**: `utils/request.ts` L73-81 + `pages/item/post.vue`

**现象**: 用户在发布页填写标题、描述、上传图片后点击发布，若此时 token 恰好过期，`request.ts` 全局拦截器检测到 401 后执行 `uni.reLaunch({ url: '/pages/auth/login' })`，页面被强制跳转，`post.vue` 的 `formData`（reactive 对象）彻底销毁。用户登录回来后需要重新填写所有信息。

**严重程度**: 高 | **影响**: 用户体验极差，尤其在填写长文本描述 + 多图上传后触发。

### 3.3 AI 审核无超时取消机制
**问题归属**: 全栈问题

**位置**: `pages/item/post.vue` L194-237

**现象**: 点击发布后按钮变为"AI 正在识别图片与校验内容..."且 `isPublishing = true`。虽然 `request.ts` 有 10s 超时，但超时后仅提示"网络通信错误"并在 `finally` 中复位 `isPublishing`。用户无法主动取消正在进行的 AI 审核请求。如果后端 AI Sidecar 实际执行了审核但网络回包丢失，用户重试可能导致重复发布。

**严重程度**: 低 | **影响**: 超时后用户需等待 10s 才能重试，无主动取消入口。

### 3.4 "想要的"统计数为硬编码 0
**问题归属**: 前端问题

**位置**: `pages/profile/profile.vue` L22-25

**现象**: 个人中心的"想要的"卡片数量始终显示为 `<text class="count">0</text>`，是静态硬编码。虽然 `detail.vue` 已实现收藏按钮且将状态持久化到 localStorage (`fav_${itemId}`)，但这个统计数字并不会从 localStorage 读取汇总。

**严重程度**: 中 | **影响**: 用户无法看到自己究竟收藏了多少物品，功能割裂。

### 3.5 无"我想要的"列表页
**问题归属**: 前端问题

**位置**: 全局

**现象**: 用户点击收藏按钮后，物品被记录在 localStorage 中，但前端没有提供查看"我收藏的物品列表"的页面入口。收藏数据成为"不可见的死数据"。

**严重程度**: 高 | **影响**: 收藏功能只完成了"写入"，未完成"读取"，对用户不具备完整使用价值。

### 3.6 聊天室历史加载失败无用户提示
**问题归属**: 前端问题

**位置**: `pages/chat/room.vue` L133-161

**现象**: `loadInitialHistory` 和 `loadMoreHistory` 的 `catch` 块中只有 `console.error`，没有任何 `uni.showToast` 或界面提示。如果 `GET /chat/history` 因为网络或后端异常而失败，聊天室会始终显示空白（"暂无消息，发送第一条问候吧"），用户不知道是"真的没消息"还是"加载失败"。

**严重程度**: 中 | **影响**: 用户可能反复进入退出聊天室，以为没有历史记录。

### 3.7 发布页图片上传无进度反馈
**问题归属**: 前端问题

**位置**: `pages/item/post.vue` L135-152

**现象**: `chooseImg` 选中图片后调用 `uni.showLoading({ title: '上传图片中...' })` 并 `Promise.all` 并发上传。如果选取 6 张大图，可能等待超过 10 秒，用户只看到转圈而没有每张图的进度。超时也仅显示"图片上传失败，请重试"，不知道哪张失败了。

**严重程度**: 低 | **影响**: 大图场景下体验不佳，但概率较低（微信会自动压缩）。

### 3.8 CSS `gap` 属性的跨端兼容风险
**问题归属**: 前端问题

**位置**: `pages/profile/published.vue` L162、`pages/index/index.vue` L491

**现象**: CSS Flex `gap` 属性在较老版本的微信小程序基础库中可能不被支持。已有先例：`detail.vue` 曾因 `gap` 导致编译错误，改为 `margin-left` 选择器解决。当前 `published.vue` 的 `.list-wrapper { gap: 20rpx }` 和 `index.vue` 的 `.loading-card { gap: 16rpx }` 仍在使用。

**严重程度**: 低 | **影响**: 在旧版微信或特定机型上卡片间距可能消失。

### 3.9 聊天消息字段大写兼容未清理
**问题归属**: 全栈问题

**位置**: `pages/chat/room.vue` L143, L147-151, L178, L200-203

**现象**: 聊天室读取消息时多处使用 `m.fromId ?? m.FromId`、`m.toId ?? m.ToId`、`m.content ?? m.Content` 等双路兜底。这表明后端 WS 消息和历史 HTTP 消息的字段名格式不一致。虽然功能不受影响，但属于代码异味，且每次读取都执行双重判断。

**严重程度**: 低 | **影响**: 代码可维护性下降，但功能正常。

### 3.10 预约页无取消按钮
**问题归属**: 前端问题

**位置**: `pages/profile/appointments.vue` + `api/item.ts`

**现象**: 后端已支持 `PUT /appointments/:id/cancel`，但预约列表页只提供了待交接状态下的"确认交接"按钮，没有"取消预约"按钮。用户无法在前端主动取消一个预约。

**严重程度**: 中 | **影响**: 用户需要通过其他渠道（如联系管理员）才能取消预约，自助性不足。

### 3.11 首页 Mock 兜底数据不完整
**问题归属**: 前端问题

**位置**: `pages/index/index.vue` L247-251

**现象**: 当基础在售流和推荐流全部失败后，首页回退到三组硬编码的 Mock 数据，其中所有物品的 `images: ['']` 为空字符串。这些物品在瀑布流中展示时会渲染缺省图 `/static/default.png`，但不会带分类标记，`publisherId` 也无对应发布者信息。

**严重程度**: 低 | **影响**: 仅在网络完全不可用时出现，但 Mock 数据本身质量较差（无图、无分类）。

### 3.12 图片 URL 拼接逻辑分散且不一致
**问题归属**: 前端问题

**位置**: `pages/index/index.vue` L320-347, `pages/item/detail.vue` L91-96, `pages/profile/published.vue` L58-87, `pages/search/search.vue` L152-159

**现象**: 四个页面各自实现了 `parseImages` + `formatImage`/`getCoverImage`，逻辑重复但细节不同：
- `index.vue` 的 `getCoverImage` 会 `.replace(/"/g, '')` 去掉 JSON 字符串残留的双引号
- `detail.vue` 的 `formatImage` 不使用 `.replace(/"/g, '')` 
- `published.vue` 的 `formatImage` 统一加 `/` 前缀
- `search.vue` 的 `getItemCover` 逻辑又稍不同

这些函数散落各处，未抽取为公共工具函数。

**严重程度**: 低 | **影响**: 修改图片处理逻辑时需改动多处，次生 bug 风险。

### 3.13 无网络状态全局感知
**问题归属**: 前端问题

**位置**: 全局

**现象**: 前端未使用 `uni.getNetworkType()` 或 `uni.onNetworkStatusChange()`。当设备完全离线时，所有请求统一走 `fail` 回调提示"网络通信错误"，但用户可能在首页、发布页、聊天室等多个地方反复触发请求，每次都会弹出相同的 Toast，且没有任何持久性的离线状态提示。

**严重程度**: 低 | **影响**: 离线体验粗糙，但 Uni-app 的 `uni.request` 本身有 native 层离线检测。

### 3.14 预约/发布页面无下拉刷新
**问题归属**: 前端问题

**位置**: `pages/profile/published.vue`, `pages/profile/appointments.vue`

**现象**: 两个列表页仅在 `onShow` 时调用加载函数，不支持下拉刷新。用户停留在列表页时若想查看最新状态，必须返回上一页再进入。

**严重程度**: 低 | **影响**: 便捷性不足，但可接受。

### 3.15 退出登录后 TabBar 状态残留风险
**问题归属**: 前端问题

**位置**: `pages/profile/profile.vue` L232-248

**现象**: 登出流程中使用 `uni.clearStorageSync()` 清空所有本地数据，但 `custom-tabbar` 仍可能持有旧的 `active` props。当 `reLaunch` 到登录页时是以 TabBar 切换之外的方式跳转，TAB 组件不会立即重新渲染。下次通过 TabBar 切换回"我的"时，个人中心页会因为 `onShow` 重新执行检查，但 `custom-tabbar` 的未读角标可能仍残留旧值。

**严重程度**: 低 | **影响**: 极短暂的状态不一致，页面刷新后恢复。

### 3.16 登录/注册参数包含密码并输出到控制台
**问题归属**: 前端问题

**位置**: `api/user.ts` L20-29

**现象**: `login()` 与 `register()` 中存在 `console.log('Sending Auth Params ...', data)`，而 `data` 包含 `studentId` 与 `password`。这会在开发者工具、真机调试日志或异常采集链路中暴露平台账号密码明文。虽然当前是平台自身密码，不是教务密码，但仍违反“密码不泄露、敏感数据最小暴露”的安全原则。

**严重程度**: 高 | **影响**: 发布环境日志可能泄露用户密码，且注册/登录是最高频入口，应优先移除或改为仅输出脱敏字段。

### 3.17 多处引用不存在的默认图片资源
**问题归属**: 前端问题

**位置**: `pages/item/detail.vue`、`pages/search/search.vue`、`pages/profile/published.vue`、`pages/profile/appointments.vue`、`pages/item/post.vue`、`pages/chat/room.vue`

**现象**: 多个页面把缺省封面回退到 `/static/default.png`，但当前 `static` 目录只有 `default-avatar.png` 与 `empty.png`，没有 `default.png`。当物品图片为空、Mock 兜底数据生效、聊天顶部物品封面缺失时，页面会请求不存在的静态资源。

**严重程度**: 中 | **影响**: 缺省图展示失败，可能在小程序控制台产生 404，影响空图、弱网和兜底场景的视觉完整性。

### 3.18 自定义导航栏未统一适配微信胶囊安全区
**问题归属**: 前端问题

**位置**: `pages/item/post.vue`、`pages/chat/room.vue`、`pages/message/message.vue`、`pages/profile/published.vue`、`pages/profile/appointments.vue`

**现象**: `search.vue` 已使用 `getMenuButtonBoundingClientRect()` 动态计算胶囊按钮与安全区域，但多个 `navigationStyle: "custom"` 页面仍仅使用 `var(--status-bar-height)` 或固定高度布局。不同设备、横竖屏或微信基础库差异下，标题、返回按钮、右侧占位可能与胶囊按钮区域接近或重叠。

**严重程度**: 中 | **影响**: 小程序真机上导航栏可点击区域和系统胶囊按钮存在冲突风险，发布页和聊天页属于核心链路，应优先统一为公共导航适配逻辑。

### 3.19 WebSocket 连接状态未暴露给页面层
**问题归属**: 前端问题

**位置**: `utils/websocket.ts`、`pages/chat/room.vue`、`components/custom-tabbar.vue`

**现象**: `utils/websocket.ts` 内部维护 `connected/retryCount/pendingQueue`，并 emit `open/close/error/message`，但 `useWebSocket()` 未导出只读连接状态，也没有页面级状态订阅示例。聊天页只能乐观渲染消息，无法根据 `close/error/reconnecting` 显示“重连中”“待发送”等状态。

**严重程度**: 中 | **影响**: 与 3.1 一致，会放大弱网场景下“看起来已发送，实际仍在队列中”的沟通误解。

### 3.20 App 级 WebSocket 监听未提供卸载或重复注册保护
**问题归属**: 前端问题

**位置**: `App.vue` L7-14

**现象**: `onLaunch` 中直接调用 `ws.on('message', anonymousCallback)`，没有保存回调引用，也没有在应用级生命周期或热更新场景中 `off`。生产小程序通常只执行一次影响较小，但 H5/开发热更新或异常重新初始化时，存在重复消费消息、未读数重复累加的风险。

**严重程度**: 低 | **影响**: 主要影响开发调试和边界生命周期，建议抽出命名回调并加注册标记。

### 3.21 发布页存在较多 Flex `gap` 兼容风险
**问题归属**: 前端问题

**位置**: `pages/item/post.vue` L291、`pages/profile/appointments.vue` L237、`pages/profile/published.vue` L162、`pages/index/index.vue` L490

**现象**: 旧报告已记录 `index.vue` 与 `published.vue` 的 `gap` 风险。本次复扫发现发布页图片网格也使用 `gap: 20rpx`，预约页列表也使用 `gap: 20rpx`。如果目标覆盖较旧微信基础库，建议将列表/网格间距统一改为子元素 margin 方案。

**严重程度**: 低 | **影响**: 旧基础库或特定机型上图片网格、预约卡片、加载卡片间距可能异常。

### 3.22 价格字段仍以金额符号强展示，需确认产品合规口径
**问题归属**: 全栈问题

**位置**: `pages/index/index.vue`、`pages/item/detail.vue`、`pages/item/post.vue`、`pages/profile/appointments.vue`

**现象**: 前端多处显示 `¥` 或“价格”字段。项目红线禁止线上支付、资金池和电商化表达，当前未出现“支付/购买/订单”等禁用词，但金额符号与“价格”在审核语境下仍可能让页面看起来像电商交易。若产品口径是“线下交接参考价”，建议 UI 文案统一为“参考价”或“期望价”，并在详情页强化“线下当面交接”。

**严重程度**: 中 | **影响**: 不一定是代码 bug，但属于微信小程序审核与平台定位风险，需要产品侧确认统一用词。

## 4. 论文撰写素材建议

### 建议一：面向弱网环境的渐进式降级架构设计
**切入点**: 本系统首页实现了"推荐流 → 基础在售流 → 本地兜底"的三级降级链，WebSocket 实现了指数退避重连与待发送队列。论文可围绕"如何在不可靠网络与 AI 服务波动下保证核心信息流可用"展开，分析每级降级的触发条件、降级文案设计原则（明确 vs 静默）、以及降级恢复策略。这体现了"韧性前端"（Resilient Frontend）的设计思想。

### 建议二：从"二元成功/失败"到"多态状态表达"的 UX 进化
**切入点**: 传统 CRUD 应用的成功/失败二元反馈已不适用于 AI 辅助系统。本系统在发布链路中设计了四种 UI 状态：AI 审核中（Loading 文案）、AI 分类修正成功、AI 分类沿用成功、AI 风控拦截失败。论文可结合状态机理论，论述如何将后端 AI 服务的不确定性（分类可能修正/可能沿用/可能不可用）转化为用户可理解的渐进式反馈，从而降低用户对 AI 系统的"黑箱焦虑"。

### 建议三：端到端行为数据采集的工程化实践
**切入点**: NCF 推荐算法依赖 `view`（浏览）、`favorite`（收藏）、`search`（搜索）三类行为埋点。论文可详细阐述三者的采集策略差异：`view` 由后端 API 自动写入（无需前端参与），`search` 在首次搜索有结果后前端静默上报，`favorite` 需要防抖锁防止连点重复。这种"后端自动 + 前端辅助 + 前端防抖"的混合采集架构，可作为论文中数据处理工程的实践章节素材。
