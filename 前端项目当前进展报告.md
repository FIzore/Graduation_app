# 前端项目当前进展报告

## 一、核心页面与交互闭环现状

### 1. 首页闭环
当前首页已经具备完整的信息流浏览能力，真实代码能力明显高于 `前端version.md` 中早期描述。

- 已实现顶部搜索入口，点击后进入独立搜索页，形成“首页浏览 -> 搜索补充探索”的自然分流。
- 已实现横向分类 Tab，分类 key 与后端 `category` 契约保持一致，当前使用 `all/book/digital/daily/sports/other`。
- 已实现分页瀑布流、触底加载、局部下拉刷新、分类切换局部 Loading、空态占位。
- 首页请求已固定走 `utils/request.ts`，不会在页面内散落原生 `uni.request`。
- 当前首页主列表请求仍使用 `GET /items`，并显式携带 `status=OnSale` 与当前分类参数。
- 当列表接口失败时，首页仍会回退到本地 Mock 数据，保证演示期 UI 不白屏。

结论：当前首页已经完成“公开流量入口”的基础闭环，但根据后端最新报告，`GET /index/recommend` 已经不是远期预埋，而是可直接接入的正式能力，因此首页推荐化改造应上调优先级。

### 2. 发布闭环
发布页已经具备可上线的基础提交流程，但 AI 感知层尚未接入。

- 已实现图片选择、并发上传、九宫格回显、删除预览、图片预览。
- 已实现标题、描述、价格、分类四类核心输入。
- 已实现发布前校验：至少一张图片、标题/描述/价格必填、分类必选。
- 已实现发布中状态锁定，按钮文案会从“发布”切到“发布中...”，并在 `finally` 中恢复。
- 发布请求已通过 `api/item.ts` 使用 camelCase 结构提交 `title/content/price/images/category`。

结论：当前发布页已经能稳定承接标准发布动作，但用户看不到后端 AI 自动修正分类、风险拦截、降级放行这三类关键结果。尤其是“AI 宕机后后端已降级放行”这一状态，前端目前完全无感知。

### 3. 详情与预约闭环
详情页已经形成从浏览到预约/私聊导流的强闭环。

- 已实现轮播图、价格、标题、状态标签、发布时间、描述区。
- 已实现预约确认弹窗、预约中 Loading、成功后本地状态切换为 `Pending`。
- 已实现“私聊卖家”入口，并在本人查看自己物品时自动隐藏。
- 当前详情页仍会在登录态下主动调用 `reportBehaviors([{ itemId, behaviorType: 'view' }])`。
- 详情页已具备基础的容错读取逻辑，兼容 `camelCase` 与历史大写字段。

结论：详情页已经可以稳定承接首页、搜索页、推荐流未来的流量落地，但根据后端最新报告，`GET /items/:id` 已经在服务端自动写入 `view` 埋点，因此前端当前主动上报更接近冗余而不是增益。

### 4. IM 与“一物一聊”闭环
聊天模块已经不是占位功能，而是具备真实业务承载能力。

- `utils/websocket.ts` 已实现全局 WebSocket 单例、待发送队列、手动关闭、指数退避重连。
- `App.vue` 会在应用启动时建立 WS 连接，并把实时消息写入 `conversationStore`。
- `conversationStore` 已按 `${otherId}_${itemId}` 建立一物一聊会话模型，避免同一用户多件物品串会话。
- `pages/message/message.vue` 已实现最近联系人列表、未读计数、按最后消息时间排序、物品标签展示。
- `pages/chat/room.vue` 已实现历史消息分页、去重、实时收发、上下文锚点卡片、查看详情回跳。
- 登出时不会粗暴清空事件监听体系，而是关闭连接、清空队列、重置会话数据，保证下次登录仍可正常复用全局监听。

结论：IM 下游承接链路已成型，这对未来“推荐流 -> 详情页 -> 私聊卖家”的 AI 导流非常关键。但后端报告也提示，`/chat/history` 是否正式依赖 `itemId` 进行隔离仍需和 API 文档完全同步，属于后续全栈核对项。

### 5. 登录态与安全登出闭环
认证与会话治理已经基本成型。

- 登录页已支持登录/注册双模式、表单校验、登录后缓存 token 与基础 userInfo。
- 登录成功后会主动调用 `ws.connect()`，避免只有冷启动才连上 WS。
- 全局请求封装已自动注入 JWT。
- `401` 已统一触发 token 清理、toast 提示、回到登录页。
- 个人中心退出时会先请求 `POST /auth/logout`，随后执行 `ws.close()`、`conversationStore.reset()`、本地缓存清理和页面跳转。

结论：当前前端已具备完整的登录态建立、失效、登出回收闭环，且与后端 HTTP 黑名单 + WebSocket 黑名单的双重失效机制一致。

### 6. 搜索闭环
搜索模块已经是独立闭环，而非首页的临时过滤器。

- 已实现独立搜索页。
- 已实现历史搜索记录、本地去重、最多 10 条保留。
- 已实现关键字搜索、结果瀑布流、触底分页、空状态提示。
- 搜索请求已经通过 `GET /items?keyword=...` 对接后端列表接口。

结论：搜索作为主动意图入口已经可用，但它目前只完成查询，没有形成后端已支持的 `search` 行为埋点闭环。

### 7. 全局容灾与降级能力
前端已经有基础容灾能力，但仍偏传统业务向。

- `utils/request.ts` 已统一处理 `401/403/429/500` 四类典型错误码。
- 首页接口失败时具备 Mock 降级能力，适合演示期保活。
- WebSocket 断连后具备自动重连与消息排队能力。
- 发布、预约等关键按钮都具备基础 Loading 恢复路径。

结论：当前容灾能力能支撑普通业务调试，但尚未针对“AI 宕机后后端已降级放行”这类部分成功状态设计可见反馈，也未针对推荐流失败提供精确降级说明。

## 二、后端 API 适配与契约同步现状

以下判断以当前真实前端代码为准，并结合 `后端项目当前进展报告.md`、`后端version.md` 与《QLUT Secondhand Platform - 生产级 API 技术规范.md》交叉校验。

### 1. 已 100% 对齐项

#### 1.1 401 认证失效
前端已完成全局 `401` 收口。

- `utils/request.ts` 会统一识别 `401`。
- 会清理本地 token。
- 会提示“登录状态已失效，请重新登录”。
- 会自动 `reLaunch('/pages/auth/login')`。

这与后端“Token 缺失/无效/已过期/已登出统一返回 401”的契约已对齐。

#### 1.2 安全登出
前端已完整接入后端 v0.26 的登出防线。

- 已调用 `POST /auth/logout`。
- 即使后端登出接口失败，前端也会继续执行本地清理，避免残留登录态。
- 登出后会关闭 WS、重置会话仓库、清空缓存、重定向到登录页。

这与后端“Token 写入 Redis 黑名单、HTTP 与 WebSocket 均立即失效”的设计方向一致。

#### 1.3 403 基础语义映射
前端已经做了最低限度的 403 语义分流。

- 登录接口 `POST /auth/login` 若返回 `403`，前端 toast 会显示“账号已锁定”。
- 其他接口若返回 `403`，前端统一显示“无权操作”。

这已经覆盖了后端锁号机制和通用越权语义的最基本感知。

#### 1.4 429 高频竞争提示
前端已经全局处理 `429`。

- 当前文案为“操作太快了，请稍后再试”。
- 这与后端在预约链路使用 Redis 锁防竞争、失败返回 `429` 的契约一致。

#### 1.5 HTTP 状态码消费方向已基本跟上后端
后端已明确从 v0.28 起切换为“真实 HTTP 状态码 + 统一 JSON 包装”。前端真实代码已基本适应这一模式。

- `utils/request.ts` 已读取 `res.statusCode` 与 `responseData.code`。
- 登录失效、限流、权限失败已经不再依赖“所有错误都包在 200 里”的旧假设。

结论：代码层已基本适应，但项目旧版版本文档仍残留“统一处理 code: 200”的表述，文档口径需要继续清理。

### 2. 部分对齐项

#### 2.1 详情页 `view` 埋点已经从“补充能力”变成“冗余风险”
这一项需要基于后端最新报告修正之前的判断。

- 后端明确说明：`GET /items/:id` 在登录态下已经自动写入 `view` 行为埋点。
- 后端还特别指出“行为数据采集不再依赖前端完全可靠上报”。
- 当前前端详情页仍额外手动调用 `reportBehaviors(view)`。

修正结论：这里不应再表述为“部分对齐、后续确认是否保留”，而应视为高优先级待清理项。继续保留前端主动 `view` 上报，收益很低，重复写入风险很高。

#### 2.2 403 语义只做了粗粒度对齐
后端把 `403` 同时用于三类场景。

- 越权操作。
- 触发风控限流或风控拦截。
- 账号已锁定。

当前前端只有两层处理。

- 登录页：账号已锁定。
- 其他页：无权操作。

结论：方向正确，但对 AI 风控、高频限制、权限不足这三类来源还没有细粒度区分，UI 解释力不足。

#### 2.3 个人中心未充分消费后端已具备的用户资产接口
当前前端虽然能展示“我发布的”和“预约交接”数量，但实现方式仍偏早期。

- `appointmentCount` 已调用 `GET /user/appointments`，契约方向正确。
- `publishedCount` 仍通过 `GET /items?page=1&pageSize=500` 拉全量列表后本地过滤发布者。
- 后端最新报告已明确“用户中心的我发布的”属于已完成的后端能力。

修正结论：这里不是“后端以后可提供专用能力”，而是“前端当前没有消费已经存在的后端资产接口”，应上调为近端重构任务。

#### 2.4 `409` 的真实实现已经先于 API 文档更新
这是本次跨端审计中最重要的文档信息差之一。

- 后端最新报告明确指出：重复注册的真实行为已经调整为 `409 Conflict`。
- 当前 API 技术规范仍写“重复注册返回 400”。
- 当前前端 `utils/request.ts` 没有 `409` 专门分支。

修正结论：之前“409 尚未纳入正式契约”的判断需要修正。更准确的说法应为：`409` 已进入后端真实实现，但 API 基准文档尚未同步，前端也尚未针对这一真实行为做精细提示。

### 3. 未对齐项

#### 3.1 首页未接入推荐接口
后端最新报告已经把 `GET /index/recommend` 定义为“已具备匿名降级与登录推荐能力”的正式后端能力，而不是仅供未来预留的空接口。

当前前端首页仍只有一套时间流列表逻辑。

- 未新增“猜你喜欢”入口。
- 未消费推荐接口。
- 未展示匿名态回退与登录态推荐差异。

修正结论：推荐流接入优先级应显著上调，已经具备进入前端首页主入口改造的条件。

#### 3.2 发布链路未体现 AI 分类覆盖结果
后端 v0.31 已支持在发布前调用 AI 侧车并自动覆盖分类。

当前前端发布页仍是纯手动分类体验。

- 用户选择分类后直接提交。
- 成功后只收到“发布成功”。
- 无法知道最终分类是否被 AI 修正。

结论：数据层可以被 AI 影响，UI 层却没有任何感知反馈。

#### 3.3 发布链路未体现 AI 风控拦截与降级放行体验
后端最新报告强调了两类前端必须感知的特殊状态。

- Python 返回高风险时，后端会拒绝发布。
- Python 宕机或超时时，后端会降级放行，不阻断核心业务。

当前前端在发布失败或成功时都没有对应语义表达。

- 没有“疑似违规图片/内容”的明确提示。
- 没有“AI 服务暂不可用，但本次已按基础规则发布成功”的说明。
- 没有区分普通发布成功和降级放行成功。

结论：后端已经支持部分成功语义，前端目前只理解传统的成功/失败二元结果。

### 4. 关于 409 的说明
本节按跨端审计后的事实口径重新修正。

- API 技术规范当前仍未正式写入 `409`。
- 但后端项目当前进展报告已明确：重复注册的真实实现已经调整为 `409 Conflict`。
- 当前前端拦截器和登录/注册页尚未对 `409` 做专门提示策略。

结论：`409` 不是“纯不存在”，而是“已存在于后端真实行为，但尚未完成文档与前端消费同步”的契约漂移项。前端需要尽快补上对注册冲突类错误的精细化提示。

## 三、AI 场景 UI/UX 预埋准备

本节直接以 `ai.md` 为锚点，并结合后端最新报告重新校准优先级。重点对照两类核心能力：CNN 智能分类感知与 NCF 推荐流展示，同时补充 AI 风控和 AI 降级反馈需求。

### 1. CNN 智能分类接入前现状
`ai.md` 明确说明：发布图片后，Go 会调用 Python AI Sidecar，对图片做 CNN 分类与风险审查。后端最新报告进一步确认：Go 主流程中的 AI 钩子、超时、轻量重试和降级放行都已就绪。

当前已具备的 UI 基础：

- 发布页已经有图片上传能力。
- 发布页已经有人工分类选择器。
- 发布页已经有提交中状态。

当前缺失的 AI 感知层：

- 缺少“AI 正在识别图片分类”的中间态 UI。
- 缺少“AI 建议分类已覆盖原选择”的结果反馈区。
- 缺少“AI 判定高风险，已阻止发布”的专属错误卡片或弹窗。
- 缺少“AI 服务暂不可用，但本次已按基础规则发布成功”的温和提示。

结论：当前发布页能完成人工分类发布，但还不能把 CNN 分类与安全风控体现为用户可理解的交互过程，也不能表达后端已经真实支持的降级放行语义。

### 2. NCF 推荐流接入前现状
`ai.md` 明确说明：推荐采用 NCF 离线训练 + Redis Top20 缓存 + 白天在线极速读取。后端最新报告进一步确认：`GET /index/recommend` 已经支持登录态读取推荐、匿名态回退最新上架、Redis 未命中再回退，不需要前端维护两套首页接口模型。

当前已具备的 UI 基础：

- 首页已经有瀑布流容器。
- 首页已经有分类 Tab 和分页加载框架。
- 详情页、私聊页、消息页已经打通下游承接链路。

当前缺失的推荐呈现层：

- 首页没有“猜你喜欢”或“为你推荐”Tab。
- 没有把推荐流与普通时间流分开。
- 没有匿名态说明，例如“当前为最新上架，登录后可查看个性化推荐”。
- 没有登录态说明，例如“根据浏览与想要记录为你推荐”。
- 没有推荐接口失败后的回退提示。

修正结论：推荐流不再只是“预留设计项”，而是应该进入前端首页近期主改造范围。

### 3. 推荐训练数据入口现状
`ai.md` 点名推荐训练依赖“浏览、收藏记录”。后端最新报告又补充：`POST /behaviors` 已支持 `view / favorite / search` 三类行为。

当前已具备：

- 后端 `view / favorite / search` 数据入口已经就绪。
- 前端搜索页已形成用户主动意图入口。
- 前端详情页目前虽然有 `view` 上报代码，但从跨端角度看该项应优先收口，改由后端详情自动埋点负责。

当前缺失：

- 没有“想要的/收藏”真实交互入口，个人中心里“想要的”仍是静态 `0`。
- 没有 `favorite` 行为上报触点。
- 搜索页没有在成功搜索时主动上报 `search` 行为。

修正结论：推荐样本入口并不是完全缺失，而是结构失衡。应优先移除重复 `view`，补齐真正缺失的 `favorite` 与 `search`。

### 4. AI 风控与 AI 降级 UX 现状
后端最新报告显示，Go 端不仅支持 AI 风控拒绝，还支持 AI 不可用时的业务降级放行。这意味着前端必须从“错误提示”进化到“状态表达”。

当前问题：

- 发布失败只有通用错误层，没有面向违规内容的专属提示。
- `403` 目前只做“账号已锁定 / 无权操作”二分，没有为 AI 风控预留用户能理解的说明口径。
- 页面没有引导用户如何修正标题、图片或描述后重新提交。
- 页面没有能力表达“AI 服务异常，但本次发布已按基础规则完成”的部分成功态。

结论：当前前端具备基础错误拦截，但还不具备“可视化 AI 风控反馈”和“可感知 AI 降级反馈”能力。

## 四、下一步任务规划 (Next Steps)

以下优先级已经根据本次跨端审计重新调整，优先处理“后端已具备而前端未消费”与“前端当前存在冗余或误判”的事项。

### [独立任务]
- 优先移除详情页手动 `view` 上报，避免与后端 `GET /items/:id` 自动埋点重复写入。
- 优先接入个人中心“我发布的”专用用户资产接口，替换当前全量 `getItems` + 本地过滤方案。
- 优先为注册页补充 `409 Conflict` 精细化提示，明确呈现“学号已注册”而不是笼统失败。
- 优先重构首页数据层，将 `GET /index/recommend` 提升为可直接接入的数据源，而不是继续停留在远期预留。

## 五、阶段结论

当前前端已经完成主业务闭环。

- 浏览、搜索、详情、预约、私聊、消息、个人中心、登录登出都已具备真实可用能力。
- 与后端既有安全防线的对齐度较高，尤其是 `401`、安全登出、基础 `403/429` 提示、真实 HTTP 状态码消费能力。
- 本次跨端审计后可以确认，前端当前最大的短板不是“缺页面”，而是三类已经成熟的后端能力还没有被正确消费：后端自动 `view` 埋点、用户资产专用接口、首页推荐正式接口。
- 同时，AI 发布链路已经具备后端降级放行能力，但前端还没有把这种“部分成功”转译成用户能感知的状态表达。

下一阶段前端工作的重点，必须从“继续补功能”转为“优先消化后端已具备的真实能力，并移除前端当前的冗余和误判”，这样 CNN 分类与 NCF 推荐才能顺利从后端预埋阶段进入真实用户体验阶段。
