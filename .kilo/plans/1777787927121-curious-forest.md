## 二、逻辑对齐：字段兼容性与 ID 匹配审计

### 2.1 camelCase / PascalCase 双重兼容：良好
所有页面模板均广泛使用 `||` 短路链做字段回退：

```
index.vue:21    :key="item.id || (item as any).ID"
index.vue:29    {{ item.title || (item as any).Title || '未命名物品' }}
detail.vue:88   item.value?.ID || item.value?.id
appointments.vue:20  app.AppointmentTime || app.appointment_time || app.CreatedAt || app.created_at
```

**结论**：运行时兼容性覆盖全面，渲染失败风险低。

### 2.2 ID 匹配 String() 转换：基本到位但存在薄弱点
- `profile.vue:211` & `published.vue:161` 均使用 `String(item.PublisherID || item.publisher_id) === currentUserId` ✅
- `profile.vue:172` 使用 `String(parsedUserInfo.ID || parsedUserInfo.id || '')` ✅
- `getUserIdFromToken()` 返回 `String(data.user_id || data.id || ...)` ✅
- `index.vue:22`：`goToDetail(item.id || (item as any).ID)` — 未做 `String()`，但作为路由参数 Uni 框架会自动序列化，`detail.vue:102` 用 `Number(options.id)` 反序列化。**无风险。**
- `detail.vue:139`：`createAppointment(itemId.value)` — `itemId` 是 `computed` 返回 `.ID || .id`，可能为 `0`（fallback）。若后端 `item_id` 为 0 会报参数错误。**低风险**。

### 2.3 乱码相关：登录页 `login.vue:76` 导入 `login` 函数时：
```
import { login, register, bindWeChat } from '../../api/user';
```
API 层 `user.ts:7` 有 `console.log('Sending Auth Params (Login):', data)` — 仅限调试用，无乱码风险。

### 2.4 防红线词汇审查：合规
代码全文检索未发现 `buy`, `pay`, `order`, `cart`, `checkout` 等违禁词。
✅ 使用 `reserve`(预约), `handover`(交接), `appointment`(预约), `pending`(锁定中)

## 四、最优先三项开发任务（按业务闭环优先级排序）

### 任务 1：WebSocket 私聊模块（对齐 Phase 4）
**影响范围**：`message.vue`(消息列表) + `utils/websocket.ts` + `pages/chat/room.vue`(聊天室)
- 创建 `utils/websocket.ts`：封装 `uni.connectSocket`，实现心跳保活、断线重连、消息分发
- 重构 `message.vue`：从占位页面改为会话列表（展示最近联系人和未读数）
- 新建 `pages/chat/room.vue`：气泡聊天界面，支持文字消息发送
- API 需补充：`/chat/history`（获取历史消息）、`/chat/conversations`（会话列表）
- **理由**：消息是 TabBar 四大入口之一，当前仅占位，是用户感知度最高的缺失功能。

### 任务 2：预约状态流转闭环（对齐 Phase 4 + API 规范）
**影响范围**：`detail.vue` + `appointments.vue`
- 在 `item.ts` 补充 `confirmAppointment` 接口（PUT /appointments/:id/confirm）
- 在 `detail.vue` 底部栏增加卖家视角判断：若当前用户 == PublisherID 且 status == 'Pending'，显示"确认交接完成"按钮
- 在 `appointments.vue` 中增加卖家侧"确认完成"操作入口
- **理由**：预约创建已实现，但缺少确认完成动作导致状态机无法闭环，这是 PRD 3.3 节的核心业务流程。

### 任务 3：搜索与空状态完善（提升 MVP 体验）
**影响范围**：`index.vue` + 新建 `pages/settings/settings.vue`
- 首页搜索栏接入 `GET /items` 的 `keyword` 参数（若后端支持）或前端本地过滤
- 创建 `pages/settings/settings.vue` 基础设置页（`profile.vue` 已引用该路由但页面不存在）
- 统一图片 BaseURL 常量化（消除各页面中重复的硬编码 `http://127.0.0.1:8080`）
- **理由**：搜索是信息撮合平台的核心交互，当前完全不可用；settings 页面缺失会导致运行时路由错误。

## 五、技术债务清单（非阻塞但建议关注）

1. **图片 URL 拼接硬编码**：4 个页面各自硬编码 `http://127.0.0.1:8080`，应抽取到 `utils/request.ts` 导出常量。
2. **TypeScript 类型不匹配**：`api/item.ts` 定义 `Item` 接口字段为 camelCase，但后端返回 PascalCase。建议定义联合类型或运行时适配层。
3. **"我发布的"数据获取低效**：`profile.vue` 和 `published.vue` 通过 `getItems({pageSize: 500})` 全量拉取后前端过滤，无后端 dedicated API。
4. **静态资源缺失**：代码中引用了 `/static/empty.png` 和 `/static/default.png`，但 `static/` 目录下仅有 `default-avatar.png`。
5. **"想要的"统计始终硬编码为 0**：`profile.vue:23` 写死 `<text class="count">0</text>`，未对接后端。
6. **详情页无"私聊"按钮**：TECH_DESIGN 3.1 要求详情页底部有"私聊"和"发起交接"两个按钮，当前仅有预约按钮。
