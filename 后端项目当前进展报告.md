# 后端项目当前进展报告

## 核心架构与业务闭环现状

截至当前阶段，Go 主网关已经完成从“课程设计原型”向“生产级业务网关”的关键演进，现有能力已经能够稳定支撑校园二手撮合平台的主业务闭环。结合 `后端version.md`、API 技术规范以及已落地的前端交互，可以确认当前后端已经具备以下生产级能力：

- 统一认证与权限体系已经成型。
  - 私有接口统一走 JWT 鉴权。
  - `POST /auth/logout` 已接入 Redis Token 黑名单，令牌退出后可立即失效。
  - `OptionalAuth()` 已落地，用于详情页与首页推荐这类“匿名可访问、登录可增强”的公开链路。
- 核心风控与并发保护已形成闭环。
  - 登录接口具备基于 Redis 的爆破锁定能力。
  - 发布接口具备场景化限流能力。
  - 预约接口使用 Redis `SetNX lock:item:<itemId>` 实现分布式锁，防止并发抢占导致的超卖。
  - 发布链路已内置本地敏感词过滤矩阵，并在 v0.31 预埋 AI 风控前置钩子。
- 即时通讯底座已经完成工程化。
  - WebSocket 鉴权、在线推送、离线落库、历史消息查询、读写超时控制、Ping/Pong 保活均已具备。
  - 黑名单 Token 在 WebSocket 场景下也已纳入校验范围，登出后不仅 HTTP 失效，长连接也会被阻断。
- 数据契约和行为闭环已具备论文级支撑能力。
  - 全站 JSON 契约已迁移至 camelCase。
  - `GET /items/:id` 支持登录态自动写入 `view` 行为埋点。
  - `POST /behaviors` 已支持 `view / favorite / search` 三类行为，为后续 Python 侧 NCF 推荐训练提供数据源。
- AI 侧车接入位已经落地。
  - Go 端已具备对 Python FastAPI Sidecar 的调用能力、超时控制、轻量重试以及故障降级策略。
  - 首页推荐接口与发布链路 AI 钩子都已在主网关侧预留完成。

从业务闭环角度看，平台当前已经完成以下主链路：

- 首页浏览与分页加载
- 搜索与分类筛选
- 物品详情查看
- 用户注册、登录、登出
- 物品发布、编辑、删除
- 预约交接、确认完成、取消恢复
- WebSocket 私聊与历史消息拉取
- 用户中心的“我发布的”和“我的预约”
- 行为埋点回流与推荐数据沉淀

这意味着 Go 主网关目前已经不是单纯的 CRUD 服务，而是已经具备“高并发保护 + 统一鉴权 + 实时通信 + 行为采集 + AI 侧车扩展能力”的业务中台雏形。对接 Python AI 微服务前，后端基础设施层面已经基本就绪。

## 跨端契约与适配现状

结合 `前端version.md` 与《QLUT Secondhand Platform - 生产级 API 技术规范.md》，当前前后端在契约层已经完成了大部分关键对齐，但仍存在少量需要继续修正的文档和接口缺口。

### 已完成的对齐

- camelCase 命名规范已基本达成一致。
  - 后端在 v0.23+ 已将 JSON 字段与 Query 参数统一为 camelCase。
  - 前端在后续版本中已经主动完成多轮对齐，明确修正了 `itemId`、`targetId`、`pageSize`、`userId`、`category` 等关键字段。
  - 当前跨端交互的主流接口模型已经不再依赖历史遗留的 snake_case 兼容写法。
- 分页逻辑已经形成一致认知。
  - 前端首页、搜索页、聊天历史页均采用 `page + pageSize + noMore` 的分页加载模型。
  - 后端 `GET /items` 与 `GET /chat/history` 文档也明确了 `page`、`pageSize` 的默认值与边界约束。
  - 后端仓储层已增加 `pageSize` 兜底，能避免异常参数导致的全表返回风险。
- Token 黑名单机制已完成前后端协同。
  - 前端 `v0.9.6` 已接入 `POST /auth/logout`，并在退出时主动清理本地登录态、会话列表与 WebSocket 连接状态。
  - 后端通过 Redis `blacklist:token:<rawToken>` 实现 HTTP 侧即时失效。
  - WebSocket 侧也已经做了黑名单校验，保证退出登录后不会继续持有可通信的旧连接。
- 详情页匿名访问与登录增强逻辑已对齐。
  - 前端已通过本地 Token 守卫避免未登录用户误触发 `/behaviors`。
  - 后端详情接口通过 `OptionalAuth()` 支持匿名访问，同时对登录用户自动补记 `view` 埋点，行为数据采集不再依赖前端完全可靠上报。

### 当前存在的潜在缺口

- 缺口 1：API 文档中的部分状态码与后端真实实现不完全一致。
  - 当前 API 文档仍写明“重复注册返回 400”，但已知后端实际行为已经调整为 `409 Conflict`。
  - 这类差异会直接影响前端精细化错误提示与测试脚本断言，应优先修正文档基线。
- 缺口 2：前端早期版本说明仍保留“统一处理 `code: 200`”的旧认知。
  - 但后端从 v0.28 起已经改为返回真实 HTTP 状态码，不再默认所有错误都走 HTTP 200 包裹。
  - 目前前端在登录失效、并发锁、业务错误提示上已经部分适应真实状态码，但项目说明文档层面仍有旧表述残留，需要统一口径。
- 缺口 3：聊天历史接口与“一物一聊”上下文之间可能仍有契约漂移。
  - 前端 `v0.9.2` 已经进入 `${otherId}_${itemId}` 的一物一聊会话模型。
  - 但当前 API 文档里的 `GET /chat/history` 只明确写了 `targetId / page / pageSize`，没有显式体现 `itemId` 会话隔离参数。
  - 如果后端历史查询在实际实现中已经依赖 `itemId` 做隔离，那么文档与前端 API 封装都还需要进一步完全同步。
- 缺口 4：首页推荐链路尚未和前端首页正式融合。
  - 后端 `GET /index/recommend` 已具备匿名降级与登录推荐能力。
  - 但前端当前首页主数据流仍以 `GET /items` 为核心，尚未正式切入推荐接口，因此“个性化推荐”在用户侧还没有完整展示闭环。
- 缺口 5：AI 风控结果尚未形成前端可感知体验闭环。
  - 当前后端已经支持 AI 分类覆盖与高风险拦截。
  - 但前端尚未明确接入“AI 拦截发布”的专属提示策略，也未对“最终分类被 AI 改写”做展示验证。

总体判断：当前前后端在基础契约层已达到“可稳定联调”的状态，尤其是在 camelCase、分页参数、鉴权与黑名单机制上已经基本对齐；剩余问题主要集中在文档漂移、聊天上下文补全以及 AI 新链路的全栈展示层接入。

## AI 侧车基建就绪度

对照 `ai.md` 中规划的两大核心创新点，即“基于 CNN 的图片智能分类与安全风控”以及“基于 NCF 的个性化推荐”，Go 侧在 v0.31 已经完成了绝大部分接入位预埋，但还没有与真实 Python AI 微服务形成最终闭环。

### 已经就绪的部分

- Go → Python 的通信基建已经具备。
  - `pkg/aiclient` 已存在，承担 Go 调用 Python FastAPI 的统一出口。
  - 当前已经具备内部 HTTP 请求封装、默认 2 秒超时、轻量重试、错误归一能力。
  - 这意味着后续 Python 服务只要按照既定契约暴露接口，Go 侧无需再做大规模结构调整。
- 发布链路的 AI 钩子已经接入主流程。
  - `POST /items` 在落库前已经可以调用 AI 分类/风控服务。
  - AI 返回 `category` 时，Go 会自动覆盖物品分类。
  - AI 返回 `riskLevel=high` 时，Go 会直接拦截发布。
  - AI 服务不可用时，Go 会优雅降级回本地发布逻辑，不阻断平台核心业务。
- 推荐链路的 Redis 读取闭环已经成立。
  - 登录用户优先读取 Redis `recommends:user:{userId}`。
  - 若命中推荐 ID 列表，Go 会回表查询 MySQL 并按 Redis Top20 顺序组装物品详情。
  - 若 Redis 未命中或用户未登录，则回退为最新在售物品流。
  - 这一设计完全符合 `ai.md` 所要求的“离线训练 + 线上极速读取”的工业化路线。
- 鉴权与推荐路由策略已满足 AI 接入需要。
  - `GET /index/recommend` 已挂载 `OptionalAuth()`。
  - 这保证了匿名用户可获得冷启动内容，登录用户可无感切入个性化推荐，不需要前端区分两套首页接口模型。

### 仍未完全闭环的部分

- Python FastAPI Sidecar 还未作为正式服务能力落地。
  - 当前完成的是 Go 端预埋和降级策略，不是 Python 服务本体的正式上线。
  - 也就是说，系统已经“预留好了插槽”，但“AI 引擎本体”尚未真正插入运行。
- CNN 分类/风控的正向联调尚未完成。
  - 目前已验证 AI 宕机时 Go 侧可降级。
  - 但还缺少真实 Python 返回 `category`、返回 `riskLevel=high`、返回非法结构时的完整联调验证。
- NCF 推荐的数据生产侧尚未打通。
  - `ai.md` 明确要求由 Python 侧基于用户埋点做离线训练，再把每个用户的 Top20 推荐结果写入 Redis。
  - 当前 Go 端已能读取 Redis，但 Python 训练任务、推荐结果写入任务、定时调度链路还未正式建立。
- AI 契约文档还不够细。
  - 当前已知 Go 侧默认路径为 `/predict/category`。
  - 但请求体格式、图片输入方式、分类枚举、风险等级枚举、错误语义、是否支持批量推理等关键细节仍需要形成正式协议文档，才能避免跨语言联调反复返工。

总体判断：AI 侧车接入并不是“从零开始”，而是已经完成了主网关侧的接口预埋、超时与降级策略、推荐缓存读取链路，属于“基建就绪、服务未完全接入”的状态。换言之，v0.31 已经把 Python AI 阶段前最难的 Go 侧改造工作提前做完了，下一阶段重点将从“主网关改造”转向“Python 服务落地与全链路联调”。

## 下一步任务规划 (Next Steps)

### [独立任务]

- 启动 Python FastAPI Sidecar 骨架搭建。
  - 明确分类/风控服务的基础目录结构、依赖管理、模型加载位置与启动方式。
  - 首批至少落地 `/predict/category` 这一核心路由，并与 Go 侧 `aiclient` 形成契约对应。
- 完成 `pkg/aiclient` 的 Mock 测试与异常路径测试。
  - 覆盖超时、拒绝连接、非 2xx、非法 JSON、返回 `category`、返回 `riskLevel=high` 等场景。
  - 目标是把当前“预埋好的调用链”从能跑提升到可回归、可证明稳定。
- 补齐 AI 内部协议文档。
  - 明确请求字段、图片输入格式、分类枚举、风险等级枚举、响应 JSON 结构、错误码及重试边界。
  - 这份文档将成为 Go、Python、前端三端联调时的唯一内部基准。
- 修复 API 基准文档与实际实现的漂移问题。
  - 同步 `/auth/register` 重复注册应为 `409`。
  - 复核 `/chat/history` 是否需要显式加入 `itemId` 参数与响应上下文说明。
  - 统一“真实 HTTP 状态码 + 统一 JSON 包装”的描述口径。
- 恢复并规范 `后端version.md` 历史版本记录。
  - 当前文件仅剩 `v0.31` 条目，不利于后续 AI 阶段持续追踪版本演进。
  - 在 Python AI 阶段正式开始前，应先修复版本追踪基线。

### [全栈任务]

- 接通首页推荐流的正式前端展示。
  - 前端首页应逐步从 `GET /items` 过渡到 `GET /index/recommend` 作为主入口。
  - 需要验证匿名用户冷启动流、登录用户推荐流、Redis 未命中降级流三种表现是否一致稳定。
- 全链路测试 AI 自动分类回填。
  - 用户在前端发布页上传图片并提交后，验证 Python 返回的 `category` 是否会被 Go 自动应用。
  - 同时验证前端最终展示的分类是否与后端实际落库一致。
- 全链路测试 AI 高风险拦截与降级表现。
  - 当 Python 返回高风险时，后端应拒绝发布，前端应给出明确 Toast 提示。
  - 当 Python 宕机或超时时，后端应降级放行，前端不应出现“误报失败”或卡死状态。
- 联调 NCF 推荐命中场景。
  - Python 离线训练将 Top20 结果写入 Redis 后，验证登录用户首页推荐顺序是否严格遵循 Redis 排序。
  - 同时验证物品被删除、下架、状态变化后，Go 侧回表组装是否仍能安全兜底。
- 复核“一物一聊”聊天上下文契约。
  - 若后端历史消息隔离依赖 `itemId`，则前端、后端、API 文档必须同步修正。
  - 这项工作应在 Python 阶段前完成，因为推荐与详情页导聊都会放大聊天上下文一致性的重要性。

## 结论

当前 Go 主网关已经完成基础业务闭环、风控与并发保护、实时通信基建以及 AI 侧车预埋，是一个可以正式承接 Python AI 微服务接入的后端底座。现阶段真正的短板不再是 Go 侧主链路，而是 Python 服务本体、AI 契约文档、推荐生产链路以及前端展示层对新能力的正式接入。

因此，项目已经具备全面进入 `ai.md` 阶段的条件，但下一步必须从“后端预埋”转向“跨语言契约落地 + Python 服务上线 + 全链路联调验证”，这样 CNN 分类与 NCF 推荐两个论文核心创新点才能真正成为系统能力，而不只是架构预案。
